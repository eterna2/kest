# Kest v0.3.0 Developer Implementation Guide

This document acts as the technical companion to the [Kest v0.3.0 Specification](./kest_spec_v0.3.0.md). Its purpose is to explain to developers **how** we intend to build each capability, the **expected behaviors** of those modules, and the underlying **design choices**.

---

## 1. Core Models & Cryptography

### **Capability 1: Standardized JWS Signatures (with FIPS alternatives)**
* **Expected Behavior**: Kest Passports must no longer use custom Base64 Ed25519 wrappers. Instead, the final output must be a standard JSON Web Signature (JWS) utilizing the `EdDSA` algorithm by default.
* **Implementation Details**:
  - **Local/Standard**: We will adopt the `PyJWT` library (with the `cryptography` backend). The payload is passed to `jwt.encode(..., algorithm="EdDSA")`. 
    - *FIPS Note*: If the host OS has a FIPS-validated OpenSSL module (e.g., RHEL FIPS mode), `cryptography` automatically conforms to FIPS 186-5 for EdDSA.
  - **Enterprise/Remote KMS (FIPS-140 Level 2/3)**: To support elite regulatory requirements, Kest provides an abstract `SignatureProvider` interface. This allows operations to inject an implementation that offloads signing logic entirely to **AWS KMS**, **Google Cloud KMS**, or **HashiCorp Vault**. This ensures hardware-backed signing without private keys ever entering Python memory space.
  - For verification, `jwt.decode(token, public_key, algorithms=["EdDSA"])` rigidly enforces checks before any OPA policies evaluate the payload.
* **Design Choice**: Moving to JWS allows any system (Node.js, Go, Rust) to verify a Kest Passport without our specific Python SDK. Providing an abstract KMS provider path ensures Kest satisfies elite enterprise compliance (FIPS) without sacrificing local developer experience.

### **Capability 2: Time-Sortable UUIDv7s**
* **Expected Behavior**: Every `entry_id` generated by Kest must be a UUIDv7, allowing chronological sorting of nodes naturally via their ID strings.
* **Implementation Details**:
  - Python's standard `uuid` library only supports v7 starting in Python 3.13. We will adopt the `uuid-utils` or `uuid6` PyPI package to generate UUIDv7s for all supported Python versions (3.9+).
  - The `KestEntry.entry_id` field will be instantiated via `str(uuid_utils.uuid7())`.
* **Design Choice**: Time-sortability is critical when hydrating massive DAGs from Deep Storage (Tier 3) or resolving concurrent Fan-In events where timestamps might collide but order of operations matters.

### **Capability 3: Node Typings & Key Discovery**
* **Expected Behavior**: The DAG must explicitly define the structural type of a node, and receivers must be able to automatically fetch public keys to verify JWS tokens using the `kid` header.
* **Implementation Details**:
  - Add `node_type: Literal["system", "data", "sanitizer", "critic", "snapshot"]` to the `KestEntry` model.
  - Implement an abstract `KeyRegistry` interface with a `get_public_key(kid: str)` method.
  - Provide a `LocalJWKStore` (dict/file based) for testing, and a `SpiffeWorkloadRegistry` or `.well-known/jwks.json` fetcher for production.
* **Design Choice**: Without a `type` literal, OPA policies cannot cleanly distinguish a passive dataset from an active agent. Without a `KeyRegistry`, JWS verification is a manual, brittle process.

## 2. Artificial Intelligence & Cognition Lineage

### **Capability: The Cognition Object**
* **Expected Behavior**: When an AI Agent processes data, it must emit a `cognition` block mapping its "reasoning" infrastructure.
* **Implementation Details**:
  - Update `KestEntry` to optionally accept a `cognition: Dict[str, Any]` field.
  - This dictionary must support standard keys: `model_profile`, `generation_config`, `system_prompt_hash`, `context_refs` (list of parent chunks), and `confidence_score`.
  - The implementation of `compute_dag_hash` in `crypto.py` must recursively hash the `cognition` block if present, binding the AI's reasoning to the cryptographic chain.
* **Design Choice**: In multi-agent systems, knowing *what* models made a decision (and their prompt versions) is as critical as knowing *who* triggered the workflow. A standardized `cognition` block ensures Critic-agents and downstream validators can explicitly query attributes like `confidence_score`.

## 3. Storage & Compaction

### **Capability: Tiered Storage (Shadow & Deep) via `fsspec`**
* **Expected Behavior**: Kest must keep HTTP headers small by offloading long histories. Developers should be able to configure `file://` local directories for CI testing and `s3://` buckets for production without changing code.
* **Implementation Details**:
  - Introduce an `FsspecStore` class inheriting from an abstract `PassportStore`.
  - The storage engine will be initialized via dependency injection at application startup (e.g., `store = FsspecStore(base_uri="s3://kest-logs/")`).
  - **The Compactor**: When `len(passport.history) >= MAX_ACTIVE_NODES`, the `FsspecStore` synchronously writes the history to `f"{base_uri}/{latest_entry_id}.json"`. It then purges the active history, leaving a single `SnapshotNode` containing `tier3_uri` and the `root_hash`.
* **Design Choice**: By adopting `fsspec`, we get support for local filesystems, Memory, S3, GCS, and Azure Blob out of the box. This fulfills the requirement for seamless local testing to production scaling.

## 4. Context Propagation

### **Capability: Implicit Passport Passing via OpenTelemetry**
* **Expected Behavior**: A developer annotating a function with `@kest_verified` should not manually need to thread the `KestPassport` through 10 nested function calls or external HTTP requests.
* **Implementation Details**:
  - Integrate with the standard `opentelemetry-api`.
  - On ingress to a verified function, we check `opentelemetry.baggage.get_baggage("kest_passport")`.
  - On egress, we update and serialize the new Passport JWS, and call `opentelemetry.baggage.set_baggage("kest_passport", new_jws)`.
  - For HTTP/gRPC boundaries, we rely on standard OTel header injectors/extractors.
* **Design Choice**: Using OTel Baggage achieves implicit context passing while relying purely on an industry standard, ensuring traces (Spans) and lineage (Kest) flow through the exact same network pipes.

## 5. Security & Policy Engine

### **Capability: Swappable OPA Engines**
* **Expected Behavior**: Developers must be able to evaluate complex Rego policies locally (with no networking overhead) and in production (via centralized OPA clusters).
* **Implementation Details**:
  - `policy.py` defines `AbstractOpaEngine` injected into the Kest Singleton.
  - **Local implementation**: `lakera-regorus` bindings. Provides blazing fast, inline Rust-based Rego execution for tight iteration loops and local tests.
    - *Fallback Note*: Because `lakera-regorus` is a compiled Rust extension with restricted OS/Python version support, the module MUST elegantly catch `ImportError`. If unavailable, Kest will instruct developers to run a local `localhost:8181` OPA docker container and automatically fall back to the `opa-python-client` for local evaluations.
  - **Remote implementation**: `opa-python-client`. Makes standard HTTP calls to a sidecar or central cluster.
  - The `payload` sent to the engine will now explicitly embed `.cognition` paths to allow security engineers to write rules like: `deny if input.cognition.confidence_score < 0.6`.
* **Design Choice**: We preserve extreme flexibility. Local execution (`lakera-regorus`) significantly lowers the barrier to entry for TDD (Test-Driven Development) since developers don't have to orchestrate Docker networks for tests.

## 6. The Trust & Taint Engine

### **Capability: Pessimistic Taint Propagation**
* **Expected Behavior**: Any risk marker (taint) introduced anywhere in the DAG must relentlessly propagate downstream to all leaves, unless formally removed by a Sanitizer.
* **Implementation Details**:
  - In the merging step (Fan-In), the `accumulated_taint` of the new `KestEntry` is calculated as the mathematical **Set Union** of all `accumulated_taint` lists from the `parent_entry_ids`.
  - The framework then explicitly appends any `added_taint` defined by the current execution.
  - *Labels* (positive guarantees) are handled inversely via **Set Intersection**—a label is only passed forward if *all* parent nodes guarantee it.
* **Design Choice**: This strict propagation enforces "Pessimistic Data Flow" and ensures that if *any* piece of PII or untrusted data entered a pipeline, the final output will definitively trigger OPA gate blocks.

### **Capability: Trust Score Decay Formula**
* **Expected Behavior**: The overall `trust_score` (0.0 to 1.0) must dynamically decay as data passes through multiple non-deterministic nodes or relies on untrusted origins.
* **Implementation Details**:
  - During node execution, Kest resolves the outgoing trust score by combining the incoming parent scores. A common, safe baseline algorithm is: `Outgoing Trust = Min([Parent Trust Scores]) * Node_Confidence`.
  - AI Agents (which emit a `cognition` object) will dynamically bridge the `cognition.confidence_score` (evaluated by the model or critic) into the node's core `trust_score`.
* **Design Choice**: Calculating trust locally at every node transition removes runtime recalculation overhead during verification. By tying the AI's `confidence_score` directly into the DAG's `trust_score` decay multiplier, downstream policy gates can seamlessly block outputs that stem from low-confidence hallucinations.

## 7. The Presentation Layer API

### **Capability: `@kest_verified` and `@kest_sanitizer`**
* **Expected Behavior**: The API boundary must abstract the cryptographic and tiering complexities from normal domain logic cleanly via Decorators.
* **Implementation Details**:
  - **`@kest_verified(added_taint=["internet"])`**: Intercepts `kwargs`, gathers context, performs Fan-In merge, runs OPA, executes domain func, appends `KestEntry`, compacted, signed, and injected back to baggage.
  - **Exception Trapping**: The `@kest_verified` and `@kest_sanitizer` decorators MUST wrap the domain function execution in a `try/except`. If an error surfaces, the execution still yields a valid `KestEntry` with the `failed_execution` taint and a zero trust score before `raise`ing the exception up the stack.
  - **`@kest_sanitizer(removes_taint="pii")`**: A specialized decorator that requires a highly authenticated identity (e.g., SPIFFE node `spiffe://internal/sanitizer`). It explicitly modifies the resulting `accumulated_taints` list by dropping the targeted taint.
* **Design Choice**: Decorators encapsulate cross-cutting concerns (logging, crypto, storage), enabling strict Single Responsibility Principle (SRP) for the actual python domain functions underneath.
